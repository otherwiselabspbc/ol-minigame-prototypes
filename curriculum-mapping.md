# Curriculum Mapping Skill

## Purpose
Use this before designing any mini-game to ensure it correctly 
targets a specific reading skill from our Lindamood Bell-informed 
curriculum. Always read this file alongside VISION.md and 
Reading_Exercise_to_Game_Mechanic.csv.

## Skill Domains (in developmental order)
1. Print Concepts
2. Letter Knowledge
3. Phonological Awareness
4. Alphabetic Principle
5. Decoding
6. Encoding

## Before building any game, identify:
1. **Target domain** — which of the 6 domains above?
2. **Target skill** — what specific skill within that domain?
3. **Input modality** — how does the child receive information?
   - Visual (sees letters, words, symbols)
   - Auditory (hears sounds, words, instructions)
   - Both
4. **Output modality** — how does the child respond?
   - Tactile (tap, swipe, trace, drag, write)
   - Speech (say, blend, repeat)
   - Both
5. **Interaction type** — what is the core mechanic?
   - Tap, Swipe, Trace, Drag, Hear+Say, See+Say, Write

## Modality rules
- Never skip the input before asking for output
  (e.g. child must HEAR sound before tapping letter)
- Always match stimulus to response type from the CSV
- Speech output requires microphone access — flag this early
- Writing/tracing requires touch or stylus — design for finger first

## Sample item requirements
- Use the exact sample items from the CSV where provided
- Real words vs nonsense words — check the CSV notes column
- Never mix skill domains in a single mini-game
- One targeted skill per game — no exceptions

## Difficulty progression within a game
- Round 1: Supported (visual + audio cues present)
- Round 2: Reduced support (one cue removed)
- Round 3: Independent (no cues, just stimulus and response)

## Red flags — stop and check with cofounder if:
- The game mechanic requires a skill the child hasn't learned yet
- Speech output is required but microphone isn't available
- You're unsure whether to use real or nonsense words
- The modality chain from the CSV isn't fully represented
